2. Instrumentación Automática

La instrumentación automática captura spans sin modificar la lógica del negocio, usando librerías que interceptan el framework HTTP y el cliente de base de datos.

Librerías usadas (requirements.txt)

opentelemetry-instrumentation-fastapi==0.45b0   # HTTP server automático
opentelemetry-instrumentation-httpx==0.45b0     # HTTP client automático
opentelemetry-instrumentation-sqlite3==0.45b0   # Base de datos automática
opentelemetry-instrumentation-logging==0.45b0   # Logs automáticos

Activación en order-service (main.py)

# Instrumenta FastAPI automáticamente: captura método, ruta, status code
FastAPIInstrumentor.instrument_app(app)

# Instrumenta httpx: propaga el contexto al llamar a kitchen-service
HTTPXClientInstrumentor().instrument()

Activación en kitchen-service (main.py)

# Instrumenta FastAPI
FastAPIInstrumentor.instrument_app(app)

# Instrumenta SQLite: genera spans por cada SELECT/INSERT
SQLite3Instrumentor().instrument()
¿Qué captura automáticamente? Cada request HTTP entrante y saliente genera un span con atributos estándar: http.method, http.route, http.status_code, http.url. Las consultas SQLite generan spans con db.system, db.statement, db.operation.

La instrumentación manual crea spans adicionales con información específica del negocio del restaurante, complementando los spans automáticos.

order.call_kitchenorder-serviceLlamada HTTP al kitchen-service

kitchen.db.query_historykitchen-serviceConsulta historial del cliente en SQLite

Ejemplo de span manual – validación de orden

# order-service/main.py – función _validate_order()
def _validate_order(order, span):
    with tracer.start_as_current_span("order.validate") as validate_span:

        # Detectar ítems no disponibles
        unknown = [i for i in order.items if i not in MENU]
        if unknown:
            validate_span.set_attribute("restaurant.unknown_items", str(unknown))
            validate_span.set_status(StatusCode.ERROR, f"Ítems no disponibles: {unknown}")
            raise HTTPException(status_code=400, detail=...)

        total = sum(MENU[i] for i in order.items)
        order_id = f"ORD-{int(time.time() * 1000)}-{random.randint(100,999)}"

        # Atributos de negocio en el span
        validate_span.set_attribute("restaurant.order_id", order_id)
        validate_span.set_attribute("restaurant.total_cop", total)
        validate_span.set_status(StatusCode.OK)
        return order_id, total

Ejemplo de span manual – consulta DB

# kitchen-service/main.py – función _query_customer_history()
def _query_customer_history(customer_name, parent_span):
    with tracer.start_as_current_span(
        "kitchen.db.query_history",
        attributes={
            "db.system":    "sqlite",
            "db.operation": "SELECT",
            "db.statement": "SELECT COUNT(*) FROM orders WHERE customer = ?",
        }
    ) as db_span:
        conn = sqlite3.connect(DB_PATH)
        cursor = conn.cursor()
        cursor.execute("SELECT COUNT(*) FROM orders WHERE customer = ?", (customer_name,))
        count = cursor.fetchone()[0]
        db_span.set_attribute("db.result_count", count)
        db_span.set_status(StatusCode.OK)
        return count

4. Propagación de Contexto

La propagación de contexto es el mecanismo que permite que el mismo traceId viaje desde el order-service hasta el kitchen-service, uniendo ambos servicios en una sola traza distribuida.

¿Cómo funciona?

OpenTelemetry usa el estándar W3C TraceContext. Cuando order-service llama a kitchen-service via HTTP, el HTTPXClientInstrumentor inyecta automáticamente las cabeceras:

traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
tracestate:  (opcional)

kitchen-service recibe esas cabeceras y extrae el contexto, continuando la misma traza con el mismo traceId.

Código que habilita la propagación (order-service)

# La instrumentación automática de httpx hace TODO el trabajo:
HTTPXClientInstrumentor().instrument()

# Al hacer esta llamada, httpx inyecta las cabeceras W3C automáticamente
async with httpx.AsyncClient() as client:
    resp = await client.post(f"{KITCHEN_URL}/kitchen/check", json=payload)
# → Header enviado: traceparent: 00-TRACE_ID-SPAN_ID-01
Resultado visible en Jaeger: Al abrir cualquier traza de order-service, se ve el waterfall completo con spans de ambos servicios bajo el mismo traceId. Esto responde directamente: "Todo debe quedar trazado de punta a punta con el mismo TraceId".

5. Correlación Logs-Traces

Cada mensaje de log incluye el trace_id y el span_id activos en ese momento, permitiendo navegar directamente de un log a su traza en Jaeger.

Configuración (ambos servicios)

# Activa la inyección automática de trace_id y span_id en logs
LoggingInstrumentor().instrument(set_logging_format=True)

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] [trace_id=%(otelTraceID)s span_id=%(otelSpanID)s] %(name)s - %(message)s",
)

Ejemplo de log generado

2026-05-17 10:23:41 [INFO] [trace_id=4bf92f3577b34da6a3ce929d0e0e4736 span_id=00f067aa0ba902b7] order-service - Orden ORD-1747478621-342 validada. Total: $57000 COP
2026-05-17 10:23:41 [INFO] [trace_id=4bf92f3577b34da6a3ce929d0e0e4736 span_id=a3ce929d0e0e4736] kitchen-service - Historial de 'Maria Lopez': 3 órdenes previas
Nótese que el trace_id es idéntico en ambos logs aunque son de servicios distintos. Con ese ID se puede buscar directamente en Jaeger y ver toda la traza.

Cómo ver los logs correlacionados

# Ver logs de ambos servicios para un trace_id específico

El ejercicio exige una métrica custom más allá de las estándar. Se implementaron dos métricas propias del negocio:

Métrica custom 1 — Valor monetario de las órdenes

# order-service/main.py
order_value_histogram = meter.create_histogram(
    "restaurant.order.value",
    description="Valor monetario de cada orden en COP",

# Se registra con cada orden aceptada:

Esta métrica permite calcular en Grafana el percentil p95 del ticket promedio por mesa, o el ingreso total acumulado del restaurante.

# Valor promedio por orden
rate(restaurant_restaurant_order_value_COP_sum[1m]) / rate(restaurant_restaurant_order_value_COP_count[1m])

Métrica custom 2 — Popularidad de platos

# kitchen-service/main.py
dish_popularity_counter = meter.create_counter(
    "restaurant.kitchen.dish_requests",
    description="Conteo de solicitudes por plato (métrica custom)",
)

# Se incrementa por cada plato en cada orden:

8. Exportación OTLP

Ambos microservicios exportan sus señales (traces, metrics) usando el protocolo OTLP sobre gRPC hacia el OTel Collector.

Configuración del exportador (ambos servicios)

OTEL_ENDPOINT = os.getenv("OTEL_EXPORTER_OTLP_ENDPOINT", "http://otel-collector:4317")

# Exportador de trazas
tracer_provider.add_span_processor(
    BatchSpanProcessor(
        OTLPSpanExporter(endpoint=OTEL_ENDPOINT, insecure=True)
    )
)

# Exportador de métricas
metric_reader = PeriodicExportingMetricReader(
    OTLPMetricExporter(endpoint=OTEL_ENDPOINT, insecure=True),
    export_interval_millis=5000,  # exporta cada 5 segundos
)

Variables de entorno en Docker Compose

    - KITCHEN_SERVICE_URL=http://kitchen-service:8001

  environment:

9. Collector Asíncrono

El OTel Collector es el componente central que recibe, procesa y enruta las señales de telemetría. Opera de forma asíncrona mediante el uso de batch processors, lo que evita que el envío de telemetría bloquee el procesamiento de los requests.

¿Por qué asíncrono?

Sin el BatchSpanProcessor, cada span sería exportado inmediatamente al Collector en el mismo thread del request (modo síncrono), añadiendo latencia directamente al usuario. Con el modo batch, los spans se acumulan en memoria y se envían en lotes, desacoplando la exportación del procesamiento de negocio.

Configuración del Batch Processor (otel-collector-config.yaml)

processors:
  # Agrupa y envía en lotes — modo asíncrono
  batch:
    timeout: 1s            # máximo 1s de espera antes de enviar
    send_batch_size: 1024  # envía cuando acumula 1024 spans
    send_batch_max_size: 2048

  # Limita uso de memoria — buena práctica SRE
  memory_limiter:
    check_interval: 1s
    limit_mib: 512         # máximo 512 MB de RAM
    spike_limit_mib: 128

  # Agrega metadatos a todas las señales
  resource:
    attributes:
      - key: environment
        value: "development"
        action: upsert

Pipelines separados por tipo de señal

service:
  pipelines:
    traces:            # trazas → Jaeger
      receivers:  [otlp]
      processors: [memory_limiter, resource, batch]
      exporters:  [otlp/jaeger, debug]

    metrics:           # métricas → Prometheus
      receivers:  [otlp, prometheus]
      processors: [memory_limiter, resource, batch]
      exporters:  [prometheus, debug]

    logs:              # logs → stdout
      receivers:  [otlp]
      processors: [memory_limiter, resource, batch]
      exporters:  [debug]
BatchSpanProcessor también en los servicios: En el código Python, BatchSpanProcessor es el que encola los spans en un buffer interno y los envía al Collector en segundo plano, sin bloquear el hilo principal de FastAPI.

12. Manejo de Errores y Spans

Cuando ocurre un error, el span queda marcado con StatusCode.ERROR y el detalle de la excepción queda registrado dentro del span como un evento.

Patrón implementado en ambos servicios

except HTTPException as exc:
    # 1. Adjunta la excepción al span (stack trace incluido)
    span.record_exception(exc)

    # 2. Marca el span con estado ERROR y mensaje descriptivo
    span.set_status(StatusCode.ERROR, str(exc))

    # 3. Incrementa el contador de errores
    orders_failed_counter.add(1, {"table": str(order.table_number)})

    # 4. Log correlacionado con el trace_id activo
    logger.error("Error procesando orden de %s: %s", order.customer_name, exc)
    raise

Tipos de error manejados

EscenarioHTTP StatusSpan afectado
Ítem no disponible en el menú400order.validate
Payload incompleto (sin campo items)422Automático FastAPI
kitchen-service no disponible500order.call_kitchen
Error en consulta SQLite500kitchen.db.*

13. Buenas Prácticas SRE

El sistema implementa las siguientes prácticas recomendadas para Site Reliability Engineering:

PrácticaImplementación
Exportación asíncrona BatchSpanProcessor y PeriodicExportingMetricReader — nunca bloquean el request
Límite de memoria Collector con memory_limiter (512 MiB) para evitar OOM
Health checks Todos los contenedores tienen healthcheck en Docker Compose
Separación de señales Pipelines independientes para traces, metrics y logs en el Collector
Atributos semánticos Uso de SpanAttributes estándar (http.method, db.system)
Logs estructurados Formato JSON-like con trace_id y span_id en cada línea
Recursos declarados service.name, service.version, deployment.environment en el Resource
Retención configurada Prometheus con retención de 7 días (--storage.tsdb.retention.time=7d)
Reinicio automático restart: unless-stopped en todos los servicios

resource = Resource.create({
    "service.name":           "order-service",
    "service.version":        "1.0.0",
    "deployment.environment": "development",
    "service.namespace":      "restaurante",
})

14. Respuestas a las Preguntas del Ejercicio

¿Dónde inició el request?
En Jaeger: el primer span de cualquier traza es order.create con service.name = order-service. El atributo http.route = /orders confirma el endpoint de entrada. El timestamp del span indica el momento exacto de inicio.
¿Cuál servicio tardó más?
En Jaeger (vista waterfall): comparar el ancho de la barra de order-service: order.create contra kitchen-service: kitchen.check. Normalmente kitchen-service tarda más porque hace 3 consultas a SQLite. En Grafana: panel "Latencia DB p95" muestra la latencia por operación.
¿Cuál span falló?
En Jaeger: filtrar con tag error=true. El span order.validate falla cuando se envían ítems que no están en el menú (ej: "sushi", "tacos"). Aparece en color rojo y contiene el atributo restaurant.unknown_items con la lista de ítems inválidos.
¿Qué operación generó latencia?
En Jaeger (waterfall): el span más ancho dentro de kitchen-service suele ser kitchen.db.query_history (consulta SQLite con WHERE sobre texto). En Grafana: panel "Latencia de Consultas DB p95" con el label operation identifica cuál de las 3 operaciones DB fue la más lenta.
¿Cuántos requests por segundo existen?
En Grafana: panel "Requests por Segundo (RPS)" usando la query:
rate(restaurant_restaurant_orders_total_total[1m])
Separado por label status (success vs failed).
¿Cuál es el percentil p95?
En Grafana: panel "Latencia de Órdenes – Percentil p95" usando:
histogram_quantile(0.95, rate(restaurant_restaurant_order_duration_ms_bucket[5m]))
Indica que el 95% de las órdenes responden en menos de ese tiempo. También se puede ver el p50 (mediana) en el mismo panel para comparar.
¿Cuál servicio consume más memoria?
En Grafana: panel "Memoria del OTel Collector" con la métrica otelcol_process_memory_rss. Para los microservicios Python, el consumo de memoria está disponible en Prometheus mediante las métricas del proceso que FastAPI expone. El Collector generalmente consume más memoria al acumular el buffer de spans del batch processor.
¿Cuál endpoint tiene más errores?
En Grafana: panel "Tasa de Error (%)" con la query:
100 * sum(restaurant_orders_failed_total) / sum(restaurant_orders_total_total)
En Jaeger: filtrar por error=true y observar cuál operation aparece más frecuentemente marcada como error. El endpoint POST /orders es el único que puede fallar cuando los ítems del pedido no existen en el menú.
